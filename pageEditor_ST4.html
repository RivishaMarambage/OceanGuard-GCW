User Profile Dashboard (profile.html)

Technical Description & Logic Engineering

The profile page was developed as a Dynamic Progress Tracker that manages user data through custom JavaScript state synchronization.

  • State Persistence: Utilized Global Scope Variables (count1, count2, count3) to maintain a persistent "Scoreboard" of user progress.This ensures that data remains consistent across different interaction modes (Initial Entry vs. Skip Correction).
  • Idempotent Validation Logic: To prevent "Double Counting" bugs, I engineered a nested conditional check: 
    if (element.innerHTML === "") { ... count++; }
    This ensures that the progress bar and fraction only increment when a field transitions from empty to filled, maintaining a strict 0-to-9 data scale.
  • Multi-Modal Feedback: The UI provides three layers of real-time feedback via the updateProgress() function:
    1. Quantitative: A CSS linear-gradient progress bar.
    2. Relative: A percentage display (e.g., 66%).
    3. Absolute: A completion fraction (e.g., 6/9) to reduce user cognitive load.
  • Conditional UI Masking: Implemented State-Dependent Visibility for the "Complete Skipped" buttons. Using the .style.display = "none" property, the buttons are programmatically hidden only once a section's specific DOM nodes are 100% validated as non-empty.
  • Visual Hierarchy: Leveraged .innerHTML for presentational markup, using <b> tags to create a clear distinction between system labels and user-provided data strings.

Testing & Validation

Extensive manual testing was conducted to verify Edge Case Handling. I confirmed that clicking "Cancel" on a prompt or providing an empty string does not trigger the increment operator, ensuring the "Guardian Rank" progress remains accurate even with erratic user input.

Sitemap (sitemap.html)

Technical Description

The sitemap was built entirely from scratch using Scalable Vector Graphics (SVG) rather than a static image or standard HTML divs. This ensures resolution independence and interactivity.

  • Coordinate System: Utilised a viewBox="0 0 1200 660" to create a responsive, mathematical grid. A strict 60px vertical rhythm was used to layout the sub-page branches.
  • Interactive Nodes: Shapes (<rect>) and text were wrapped in standard <a> tags to create clickable website navigation directly inside the SVG graphic.
  • CSS Styling: Implemented a hierarchical color scheme using CSS classes (.node-parent, .node-child) to represent different levels of website depth visually. Hover transitions were added to improve user feedback.
  • Structural Layering: Lines were coded before shapes to take advantage of the SVG painter's algorithm, ensuring connecting branches rendered cleanly behind the interactive buttons.

Accessibility

Non-clickable category headers were wrapped in <g> (group) tags rather than anchor links to prevent user confusion. The container was styled with overflow-x: auto to ensure horizontal scrolling on mobile devices rather than breaking the layout.

🧠 Challenges and Lessons Learned

1. Mapping the Sitemap (SVG Grid Planning)

The most difficult visual task was creating the Sitemap from scratch using SVG code. Because I wasn't using a simple image, I had to plan the exact position of every box and line on a mathematical grid.

  • The Problem: At first, the lines connecting the pages didn't touch the boxes perfectly, leaving small gaps at the corners that looked unprofessional.
  • The Solution: I researched how SVG lines work and used a property called stroke-linecap: round. This smoothed out the joints where lines met, making the whole map look like one solid structure.
  • Lesson: I learned how to use the CSS Box Model and Overflow properties to make sure this large map didn't break the layout on small mobile screens.

2. Fixing the Progress Bar (The "Over-Counting" Bug)

A major logic error happened while building the User Profile. If a user clicked the "Complete Skipped" button more than once, the progress bar would keep adding points forever, eventually going way past 100%.

  • The Problem: The code was just adding +1 every time the button was pressed. It wasn't "smart" enough to check if the information was already there.
  • The Solution: I changed the code to check the screen first. Now, the computer looks at the display box to see if it is empty. It only adds a point if the box is actually blank. This makes the counting "self-correcting."
  • Lesson: This taught me Defensive Programming—writing code that doesn't break even if a user clicks things in the wrong order or multiple times.

3. Better Progress Feedback (Percentage and Fractions)

I realized that just having a percentage bar (like 33%) wasn't helpful enough for users because they didn't know exactly how many questions were left to answer.

  • The Problem: Percentages can be vague. Users often feel "estimation fatigue" when they can't see the exact number of steps remaining.
  • The Solution: I updated my updateProgress() function to show two things at once: the percentage and a fraction (like 3/9). I also made the "Complete Skipped" button disappear automatically, but only when a section is truly finished.
  • Lesson: I learned that good web design is about Visibility of System Status—always making sure the user knows exactly where they stand in a process.

♿ Accessibility

  • Alt text: Every <img> element includes descriptive alt attributes that convey the image content and context.
  • Semantic HTML: Pages use <header>, <nav>, <main>, <footer>, and <article> elements for proper document structure.
  • Visual Hierarchy: The sitemap uses a distinct cascading color palette to communicate depth and structure intuitively.
  • Readable contrast: Dark text is used on light backgrounds to ensure contrast ratios exceed WCAG AA requirements.
  • Responsive Containers: Large diagrams like the sitemap are contained within scrollable divs to prevent layout breaking on mobile.
